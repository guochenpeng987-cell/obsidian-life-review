# obsidian-life-review 使用说明

> 建议放置路径：`06 系统/04 规则/obsidian-life-review 使用说明.md`
>
> 文档定位：说明这套仓库该怎么用、日常流程怎么跑、审计怎么看、ChatGPT 怎么接入。

## 一、这套仓库是什么

`obsidian-life-review` 不是日常主写作仓库，而是你 Obsidian 人生系统的**审阅镜像**。

它的主要作用有 4 个：

1. 保留一级目录和主要模块结构。
2. 把 Markdown 暴露给 ChatGPT 做持续审阅。
3. 通过 `_audit/current/` 提供更容易阅读的审计视图。
4. 让夜审 / 周审可以稳定复用，而不是每次从头解释系统结构。

所以这套仓库的第一目标不是“高频编辑”，而是：

- 可审阅
- 可回看
- 可发现结构问题
- 可支持小步优化

日常主编辑仍应在本地 Obsidian Vault 完成；这个仓库更像**审阅外壳**、**镜像窗口**和**系统体检面板**。

## 二、推荐阅读顺序

每次要审阅仓库，建议按下面顺序进入：

1. [README.md](../../README.md)
2. [_audit/current/00_tree.md](../../_audit/current/00_tree.md)
3. [_audit/current/01_inventory.csv](../../_audit/current/01_inventory.csv)
4. [_audit/current/02_core_notes.md](../../_audit/current/02_core_notes.md)
5. [_audit/current/03_changed_notes.md](../../_audit/current/03_changed_notes.md)
6. [_audit/current/04_link_health.csv](../../_audit/current/04_link_health.csv)
7. 如有必要，再回到原始笔记看上下文

这套顺序的意义是：

- 先看**全局结构**
- 再看**全库台账**
- 再看**系统骨架**
- 再看**最近变化**
- 最后看**结构健康风险**

不要一上来就随机点开笔记。那样看到的是局部，而不是系统。

## 三、一级目录怎么用

### 1. `00 收件箱`

用途：临时接住内容，不做长期存放。

适合放：

- 临时网页链接
- 一时想到但还没分类的想法
- 待处理的文章标题
- 手机/网页随手收下来的内容

使用原则：

- 这里只负责“接住”，不负责“长期保留”。
- 默认只记最小信息：标题、链接、为什么先收下它。
- 24–48 小时内必须做一次初判：留下、转入知识、挂到项目、或删除。

不要把收件箱用成长期仓库。

### 2. `01 日记`

用途：每日记录与周期回顾。

它承担的不是“记流水账”，而是把当天的：

- 重点
- 项目推进
- 输入
- 输出
- 身心状态
- 晚间复盘

放到同一张页面里。

正确用法是：

- 当天读到的重要内容，挂到当日日记
- 当天推进的项目，挂到当日日记
- 晚上复盘时，把“今天真正发生的变化”写出来

这样日记就不是孤立页面，而是系统里的**时间轴**。

### 3. `02 领域`

用途：长期责任区与维持标准。

领域不是“我感兴趣的主题”，而是你必须长期维护的现实责任区。

当前系统已经按领域拆成 8 个方向，每个方向原则上应有：

- 一个领域主页
- 若干子主题或支撑页面
- 清晰的维持标准
- 当前阶段重点
- 必要的关键指标

领域页适合承载：

- 这个领域为什么重要
- 当前阶段最需要维持什么
- 什么状态算失衡
- 未来 1–3 个月的重点是什么

不要把领域页写成资料堆或链接仓。

### 4. `03 项目`

用途：阶段性推进事项。

项目的特点是：

- 有明确结果
- 有时间跨度
- 有完成标准
- 需要推进记录

项目页应该尽量保持轻而清晰，至少能一眼回答：

- 这个项目要做成什么
- 现在处于什么状态
- 下一步是什么
- 当前阻塞点是什么

项目页不是模板收藏区，也不是资料垃圾桶。

### 5. `04 知识`

用途：输入后的加工区。

知识库不应主要靠深层文件夹运转，而应主要靠：

- 属性
- 链接
- 搜索
- 主题聚合
- 视图筛选

当前实际已落地的知识层包括：

- `01 摘录`
- `02 主题笔记`
- `00 知识总览.md`
- `05 主题地图.md`

规则层已经定义了 4 个知识归宿：

1. 摘录
2. 主题笔记
3. 长青笔记
4. 方法清单

因此后续扩建知识库时，优先考虑补齐：

- `04 知识/03 长青笔记`
- `04 知识/04 方法清单`

不要继续横向扩更多主题文件夹；优先让“成熟度”变清楚。

### 6. `05 人物`

用途：关系管理。

人物页的价值不在“收集联系人”，而在：

- 记录关系定位
- 记录关键偏好
- 记录最近互动
- 记录后续待跟进事项
- 把人物与项目 / 会议 / 决策连起来

当人物数量还不多时，先平铺即可，不必过早分太多层。

### 7. `06 系统`

用途：模板、导航、规则、说明。

它是这套系统的控制层，包含：

- 模板
- 导航
- 使用规则
- 系统说明
- ChatGPT 提示词

这里的内容应该尽量稳定、可复用、少冗余。

### 8. `99 归档`

这是设计上已经预留的模块，用来放：

- 已结束项目
- 低频但仍需保留的材料
- 不再活跃但不能删除的页面

如果当前仓库还没正式创建这一层，可以先不补；等系统真的出现稳定归档需求时再启用。

### 9. `_audit/current`

用途：系统体检面板。

这里不是手写区，而是**生成区**。它负责展示当前镜像的：

- 目录树
- 台账
- 核心骨架
- 最近变动
- 链接健康状态

不要手改这里的内容；应通过脚本重新生成。

### 10. `scripts`

用途：生成镜像与审计包。

这个目录下的脚本负责：

- 从本地 Vault 读取受管 Markdown
- 同步到 review mirror
- 生成 `_audit/current/` 下的审计文件

你日常不是“维护脚本输出”，而是“维护源笔记 + 运行脚本”。

## 四、日常使用最小闭环

这套系统最小可运行闭环只有 5 步：

1. 先接住
2. 再初判
3. 再加工
4. 再连接
5. 再复盘

### 第一步：先接住

看到值得留下的内容时，不要一开始就想“它最终放哪里”。

默认动作：

- 先进入 `00 收件箱`
- 只留下最小必要信息

最小信息通常够这 3 项：

- 标题
- 原始链接
- 为什么先收下它

### 第二步：再初判

当天晚上或次日，至少问自己 3 个问题：

1. 这条内容在帮我解决什么现实问题？
2. 它属于哪个领域？
3. 它更像资料、方法，还是原则？

这一步的目标不是写长笔记，而是判断它值不值得进入正式系统。

### 第三步：再加工

如果值得入库，就用合适模板把它加工成标准笔记。

最常见入口是：

- [文章摘录模版](../01%20模板/文章摘录模版.md)
- [主题笔记模版](../01%20模板/主题笔记模版.md)
- [长青笔记模版](../01%20模板/长青笔记模版.md)
- [方法清单模版](../01%20模板/方法清单模版.md)

### 第四步：再连接

一篇正式笔记进入系统后，不应孤立存在。

高价值内容至少连到一个“活页”之一：

- 领域页
- 项目页
- 当日日记
- 人物页

没有被连接的好内容，很容易变成“存过但不会再用”的沉没笔记。

### 第五步：再复盘

每天至少做夜审；每周至少做一次周审。

目标不是一次性重构系统，而是：

- 找 1–3 个最值得动的点
- 做小步高收益优化
- 让结构和内容慢慢贴合真实使用

## 五、文章进入系统的标准流程

与文章相关的正式规则，请优先参考：

- [文章收录及使用规则](./文章收录及使用规则.md)
- [知识采集栈](./知识采集栈.md)
- [文章收录提示词](./人生系统常用提示词/文章收录提示词.md)

这里给出最常用的实际版流程。

### 1. 默认先进收件箱

看到一篇好文章时，默认先进 `00 收件箱`。

原因很简单：

- 这样最轻
- 最不容易过度处理
- 最适合先接住再判断

### 2. 24 小时内做初判

把文章从收件箱拿出来时，按下面顺序判断：

#### 这篇文章在帮我解决什么问题？

例如：

- 求职
- 面试
- 精力恢复
- 财务规划
- 写作表达
- 沟通协作

#### 它属于哪个领域？

必须挂到一个最主要的领域，而不是无限多挂。

#### 它最后更像什么？

这一步决定去向：

- 只是资料：进摘录
- 属于某个持续主题：进主题笔记
- 提炼出长期有效原则：进长青笔记
- 能直接照着做：进方法清单

### 3. 四种标准去向

#### A. 摘录

适合：

- 文章值得保留
- 但还不值得提炼成长期原则
- 或当前只需要先留下原材料

放置位置：

- `04 知识/01 摘录`

#### B. 主题笔记

适合：

- 同一主题已经积累了多篇相关内容
- 需要从“文章堆”升级为“专题页”

放置位置：

- `04 知识/02 主题笔记`

#### C. 长青笔记

适合：

- 内容已提炼成可长期复用的原则
- 重点不再是原文，而是你自己的表达

建议放置位置：

- `04 知识/03 长青笔记`

#### D. 方法清单

适合：

- 内容可以直接变成步骤、流程、检查清单
- 未来需要反复执行

建议放置位置：

- `04 知识/04 方法清单`

### 4. 进入知识库后仍要继续连接

如果文章和当前项目直接相关，顺手挂到项目页。

如果文章和某个长期领域强相关，顺手挂到领域主页。

如果文章影响了你今天的判断或动作，顺手挂到当日日记。

记住：**文章不是收藏品，而是原材料。**

## 六、各类笔记怎么用

### 1. 收件箱条目

用途：临时接住。

应该包含：

- 标题
- 链接
- 一句话收藏原因

不要写成：

- 长摘要
- 长笔记
- 永久存档

### 2. 文章摘录

入口模板： [文章摘录模版](../01%20模板/文章摘录模版.md)

适合场景：

- 内容值得保留
- 还处在加工前期
- 需要把一篇文章变成结构化笔记

核心目标：

- 说清楚为什么留下
- 用自己的话概括核心内容
- 标出可执行动作
- 判断最终去向

不要写成：

- 纯链接保存页
- 全文复制页
- 没有“对我有什么用”的空摘录

### 3. 主题笔记

入口模板： [主题笔记模版](../01%20模板/主题笔记模版.md)

适合场景：

- 同主题内容出现多次
- 你需要一个专题工作台

核心目标：

- 汇总不同来源观点
- 形成自己的主题框架
- 为后续行动或表达服务

不要写成：

- 一篇文章对应一张“伪主题页”
- 只是罗列链接，不做归纳

### 4. 长青笔记

入口模板： [长青笔记模版](../01%20模板/长青笔记模版.md)

适合场景：

- 提炼出长期有效的原则
- 希望以后能反复调用

核心目标：

- 让自己的话占主体
- 把外部内容翻译成自己的判断框架

不要写成：

- 文章摘要复刻版
- 只换标题不换内容的假长青

### 5. 方法清单

入口模板： [方法清单模版](../01%20模板/方法清单模版.md)

适合场景：

- 内容已能直接执行
- 未来需要重复使用

核心目标：

- 形成步骤
- 形成检查项
- 形成适用场景
- 形成完成标准

不要写成：

- 空洞原则页
- 没有动作的抽象说明

### 6. 项目页

入口模板： [项目模版](../01%20模板/项目模版.md)

适合场景：

- 有明确结果与阶段目标
- 需要跨天推进

项目页至少要看得见：

- 目标
- 当前状态
- 下一步动作
- 关键资料
- 关键决策
- 推进记录

不要写成：

- 模板残留页
- 资料堆积页
- 没有下一步的“假项目”

### 7. 领域主页

入口模板： [领域首页模板](../01%20模板/领域首页模板.md)

适合场景：

- 需要定义长期维持标准
- 需要明确当前阶段关注点

核心目标：

- 回答“这个领域为什么重要”
- 回答“当前最该维持什么”
- 回答“什么情况算失衡”

不要写成：

- 资源目录
- 资料堆
- 模板示例页

### 8. 人物卡

入口模板： [人物卡模板](../01%20模板/人物卡模板.md)

适合场景：

- 需要持续互动的人物
- 会和项目、协作、关系维护相关的人物

核心目标：

- 记录偏好
- 记录互动历史
- 记录待跟进事项
- 连到项目 / 会议 / 决策

不要写成：

- 通讯录镜像
- 只有姓名没有关系上下文的空卡片

### 9. 每日笔记

入口模板： [每日笔记模板](../01%20模板/每日笔记模板.md)

核心目标：

- 把当天重点、推进、输入、输出、状态放到一张页里
- 成为夜审时最可靠的“当天证据”

### 10. 每周回顾

入口模板： [每周回顾模板](../01%20模板/每周回顾模板.md)

核心目标：

- 不重复日记内容
- 提取一周内真正反复出现的问题、进展和偏差
- 给下一周做选择，而不是做抄录

## 七、采集源与知识采集栈怎么配合

与采集相关的核心页包括：

- [采集源总览](../03%20导航/采集源总览.md)
- [知识采集栈](./知识采集栈.md)

二者分工要分清：

### `采集源总览`

负责记录：

- 我订了哪些来源
- 为什么订
- 当前状态如何
- 什么时候复盘

它是**来源台账**，不是文章存放区。

### `知识采集栈`

负责规定：

- 采集原则
- 采集范围
- 工具栈
- 采集节奏
- 入库条件
- 清理节奏

它是**输入操作手册**，不是具体文章笔记。

正确分工应是：

- 来源台账告诉你“从哪里来”
- 采集规则告诉你“怎么进”
- 知识笔记告诉你“怎么加工”
- 领域 / 项目 / 日记告诉你“怎么用”

## 八、审计文件怎么读

### 1. `00_tree.md`

看什么：

- 层级是否清晰
- 是否有重复层
- 是否有空层
- 是否有职责混乱

适合回答：

- 结构是不是越长越乱
- 目录有没有铺过头

### 2. `01_inventory.csv`

看什么：

- 路径
- 类型
- 修改时间
- 字数估算
- 标签
- 链接进出情况
- 空笔记 / 重复标题等指标

适合回答：

- 哪个模块过重或过轻
- 哪些文件过空
- 哪些文件命名有问题

### 3. `02_core_notes.md`

看什么：

- 首页
- 总览页
- 领域主页
- 规则页
- 系统说明

适合回答：

- 系统骨架是不是在同一套逻辑下
- 规则和实际结构是否脱节

### 4. `03_changed_notes.md`

看什么：

- 最近 24 小时内真正变动过的笔记

适合回答：

- 今天加的东西放对位置了吗
- 最近是否又长出模板残留
- 今晚最应该处理哪几页

### 5. `04_link_health.csv`

看什么：

- 孤儿笔记
- 断链
- 空笔记
- 重复 stem
- 链入 / 链出情况

适合回答：

- 哪些内容有失联风险
- 哪些示例链接和占位模板在制造噪音

## 九、脚本怎么跑

### 1. 准备配置

先在本地准备 `config.json`。

如果仓库里只有示例配置，先复制一份再改自己的路径。

配置至少要能回答两件事：

- 本地 Vault 在哪里
- review mirror 仓库在哪里

### 2. 运行入口

默认入口是：

- [scripts/run_audit.bat](../../scripts/run_audit.bat)

它的作用是：

1. 检查 `config.json` 是否存在
2. 调用 `scripts/build_audit_pack.py`
3. 生成镜像与 `_audit/current/` 审计文件
4. 提醒你执行 `git add / commit / push`

### 3. 构建脚本在做什么

生成脚本：

- 扫描受管 Markdown
- 解析 frontmatter
- 同步镜像文件
- 统计字数、标签、链接进出
- 识别断链、孤儿、重复 stem、空笔记
- 生成 5 份审计文件

因此有两个原则必须记住：

- **不要手改 `_audit/current/` 下的文件**
- **不要把审计输出当作人工维护内容**

### 4. 正确更新流程

一次标准更新流程如下：

1. 在本地 Vault 里完成真实修改
2. 运行 `scripts/run_audit.bat`
3. 检查 `_audit/current/` 是否生成正常
4. 提交并推送仓库
5. 再让 ChatGPT 读取最新仓库做夜审 / 周审

## 十、ChatGPT 怎么接进来

### 1. 处理文章

处理文章时，优先使用：

- [文章收录提示词](./人生系统常用提示词/文章收录提示词.md)

推荐做法：

- 把文章链接发给模型
- 要求模型完整阅读
- 按你的摘录模板输出 Markdown
- 再手动检查并落库

目标不是“听一段总结”，而是直接得到**可进系统的结构化笔记**。

### 2. 做夜审 / 周审

审阅入口是：

- [REVIEW_START.md](../../REVIEW_START.md)

其中已经写好了：

- 夜审提示词
- 周审提示词
- 使用建议

推荐节奏：

- 每晚优先看 `03_changed_notes.md`
- 每周再结合 `00_tree.md + 01_inventory.csv + 02_core_notes.md` 做全量审阅

### 3. 对 ChatGPT 的正确预期

ChatGPT 更适合做：

- 结构检查
- 命名审查
- 模块失衡判断
- 空模板 / 断链 / 孤儿风险识别
- 小步高收益优化建议

不要期待它一口气替你完成整个系统重构。

正确问法是：

- 先给当前仓库
- 再给明确审阅顺序
- 最后只要 3–5 个最值得做的动作

## 十一、当前版本使用时要注意的现实问题

这一节是给“当前版本”看的提醒，后续如果结构补齐，可以删掉或改写。

### 1. 导航页未必已经是最终总控台

如果 `06 系统/03 导航/人生系统-首页.md` 仍处在占位状态，就不要把它当成唯一入口。

### 2. 知识层规则已经比目录走得更快

规则里已经有：

- 长青笔记
- 方法清单

但目录如果还没真正落地，就应尽快补齐，避免“规则存在、落点缺失”。

### 3. 模板和正式笔记要严格分离

模板只放在：

- `06 系统/01 模板`

正式笔记进入领域 / 项目 / 知识 / 人物模块前，必须先删掉：

- 占位标题
- 示例链接
- 空白章节
- 重复模板段落

### 4. 标签与 frontmatter 口径要尽量统一

后续如果要持续依赖审计脚本、Dataview 或 Bases，frontmatter 的字段命名要尽快统一。

尤其是：

- 日期格式
- 标签字段
- 类型字段
- 状态字段

口径统一越早，后面越省事。

## 十二、新增一条内容时的决策清单

每次新内容进入系统前，至少问这 7 个问题：

1. 它在帮我解决什么现实问题？
2. 它属于哪个主要领域？
3. 它和当前哪个项目有关？
4. 它值不值得进入 Obsidian，而不是只留在阅读器 / 浏览器？
5. 它应该是摘录、主题笔记、长青笔记，还是方法清单？
6. 它进入系统后，至少要连到哪一个活页？
7. 如果 24–48 小时内还没处理，我是该删掉，还是明确归位？

这 7 个问题比“先把内容留下来再说”更重要。

## 十三、三条执行底线

### 1. 收件箱不长期堆积

收件箱只负责接住，不负责长期保存。

### 2. 正式笔记不保留模板残留

只要进入正式模块，就必须清掉占位标题、示例链接、空章节和重复模板段落。

### 3. 高价值笔记至少连到一个活页

好的知识如果不连到：

- 领域
- 项目
- 日记
- 人物

它大概率就只是“存过”。

## 十四、建议从这里开始用

如果今天要重新开始跑这套系统，最小动作就是：

1. 用 `00 收件箱` 接住今天看到的内容
2. 晚上选 1 篇文章做成摘录
3. 把这篇摘录至少连到 1 个领域页或项目页
4. 运行一次审计脚本
5. 用 `REVIEW_START.md` 做一次夜审

这样做的目标不是“立刻完整”，而是先把闭环跑起来。
